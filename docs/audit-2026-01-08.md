**Date:** 2026-01-08
**Scope:** Complete codebase analysis (92 .nix files, 377 unique packages)

---

## Executive Summary

The sinnix configuration is well-structured with clear modular organization, but has accumulated technical debt through:
- **54 package duplications** across modules  
- **Redundant tool categories** (4 git UIs, 3 CSV tools, 3 calculators)
- **Commented-out packages** indicating build failures not cleaned up
- **Settings redundancy** (GC configured in 2 places, locale settings repetitive)
- **Refactoring opportunities** in path management and service configuration

**Estimated cleanup potential:** ~20-30 packages removable, ~15% LOC reduction possible

## 4. Settings Redundancy

### Garbage Collection Configured Twice
- **System level:** `modules/core.nix:62-66`
  ```nix
  nix.gc = {
    automatic = true;
    dates = "weekly";
    options = "--delete-older-than 30d";
  };
  ```
- **Home Manager:** `modules/features/cli/core.nix:108-112`
  ```nix
  nix.gc = {
    automatic = true;
    dates = "weekly";
    options = "--delete-older-than 30d";
  };
  ```

**Issue:** Both are identical but configured separately.  
**Action:** Remove from cli/core.nix, system-level GC is sufficient

### Locale Settings Repetition
In `modules/core.nix:101-111`, all `LC_*` variables are set to `pl_PL.UTF-8`:
```nix
LC_ADDRESS = "pl_PL.UTF-8";
LC_IDENTIFICATION = "pl_PL.UTF-8";
# ... 7 more identical settings
```

**Action:** Simplify using `lib.genAttrs`:
```nix
extraLocaleSettings = lib.genAttrs [
  "LC_ADDRESS" "LC_IDENTIFICATION" "LC_MEASUREMENT" 
  "LC_MONETARY" "LC_NAME" "LC_NUMERIC" 
  "LC_PAPER" "LC_TELEPHONE" "LC_TIME"
] (_: "pl_PL.UTF-8");
```

### Systemd Tmpfiles Path Hardcoding
Lines 148-179 in `core.nix` hardcode paths that should reference `config.sinnix.paths.*`:

**Current:**
```nix
systemd.tmpfiles.rules = [
  "d ${paths.outerRealm}/inbox 0755 ..."
  # ... then later:
  ++ lib.optional (paths.realmRoot or "" == "/realm") "d /realm/inbox ..."
];
```

**Issue:** Path duplication and hardcoded "/realm" strings.  
**Action:** Use path config consistently, remove conditional overrides

### Unnecessary mkDefault/mkForce Usage

Several `lib.mkDefault` or `lib.mkForce` usages may be unnecessary:

**core.nix:40:** `max-jobs = lib.mkDefault 6;`  
**core.nix:79:** `hostPlatform = lib.mkDefault "x86_64-linux";`  
**core.nix:43:** `gnome.gnome-keyring.enable = lib.mkForce false;`  

**Action:** Audit whether these priority overrides are needed or if direct assignment suffices

## 5. Code Quality & Refactoring Opportunities

### A. Path Management Centralization

**Current state:** Paths defined in foundation.nix but referenced inconsistently
- Some modules use `config.sinnix.paths.capturesRoot`
- Others hardcode `"/realm/data/captures"`
- Tmpfiles rules mix both approaches

**Recommendation:**
- Create `lib.sinnix.mkPath` helper for path derivation
- Audit all path references for consistency
- Remove hardcoded "/realm" strings

### B. Feature Module Boilerplate

**Pattern observed:**
```nix
{
  options.sinnix.features.{category}.{name}.enable = lib.mkEnableOption "...";
  config = lib.mkIf cfg.enable { ... };
}
```

**Current:** `mkFeatureModule` helper exists but not universally used

**Action:** Refactor all feature modules to use `mkFeatureModule` consistently

### C. Service Module Duplication

Both `services/asciinema.nix` and `features/cli/asciinema.nix` exist for asciinema.

**Recommendation:**
- Consolidate into single module in features/
- Or clearly separate: service (daemon) vs. cli tool (package)

### D. Browser Module Scope Creep

`modules/features/desktop/browser.nix` contains:
- Browser packages (qutebrowser, chrome, firefox, tor)
- **BUT ALSO:** Network tools (nmap, tcpdump, wireshark, mtr)
- **AND:** SSH configuration
- **AND:** Generic tools (curl, wget, fzf, jq)

**Recommendation:**
- Move network tools to dedicated `features/desktop/networking.nix` or `dev/networking.nix`
- Move SSH to `features/cli/ssh.nix`
- Keep only browser-specific packages in browser.nix

### E. Python Package Management

**Current:** Monolithic Python environment in `dev/languages.nix:37-72` (35 packages)

**Issues:**
- All Python packages installed globally
- No project-specific environments
- Heavy base environment

**Recommendation:**
- Split into:
  - **Base:** Essential tools (pip, ipython, rich, click, typer, tqdm)
  - **AI:** LLM libraries (openai, anthropic, litellm, tiktoken)
  - **Data:** Data tools (pydantic, httpx, beautifulsoup4)
  - **Dev:** Development tools (black, mypy, pytest)
- Consider moving AI tools to a separate feature module
- Use devenv for project-specific dependencies

### F. Overlay Organization

12 overlay files in `flake/overlay/package/` with inconsistent patterns:

**Recommendation:**
- Group related overlays (python packages together)
- Add overlay documentation header explaining purpose
- Consider splitting `default.nix` into category files

## 6. Process for Detecting Unused Packages

### Proposed Multi-Stage Process:

**Stage 1: Static Declaration Analysis (Done)**
- Extract all declared packages from .nix files
- Cross-reference declarations across modules
- Identify duplicates and commented-out packages

**Stage 2: Runtime Usage Verification** (Recommended)
```bash
# For each declared package:
# 1. Check if binary exists in $PATH after rebuild
# 2. Scan shell history for usage
# 3. Check systemd service ExecStart commands
# 4. Grep dotfiles for references

# Example script:
for pkg in $(cat all-packages.txt); do
  # Check if in PATH
  command -v "$pkg" >/dev/null 2>&1 || echo "NOT FOUND: $pkg"
  
  # Check shell history  
  grep -q "$pkg" ~/.zsh_history || echo "NOT IN HISTORY: $pkg"
  
  # Check systemd services
  systemctl --user --all | grep -q "$pkg" || echo "NOT IN SYSTEMD: $pkg"
done
```

**Stage 3: Gradual Removal Testing**
1. Mark candidate packages with `# TODO: Verify usage` comment
2. Create feature branch removing candidates
3. Test for 1-2 weeks
4. Monitor for missing command errors
5. Restore if needed, otherwise merge

**Stage 4: Continuous Monitoring** 
- Add pre-commit hook checking for duplicate package declarations
- Monthly audit of package count trends
- Track package additions in git commit messages

## 9. Conclusion

The sinnix configuration is architecturally sound with excellent modular organization. The primary issues are:

1. **Accumulation of duplicates** from feature module expansion
2. **Tool redundancy** from trying multiple similar tools
3. **Incomplete cleanup** of broken/experimental packages
4. **Settings configured in multiple places** (GC, paths)

**These are normal evolution patterns** and easily addressable through systematic cleanup.

**Recommendation:** Implement immediate fixes (1-5) this week, short-term (6-10) over next month, long-term as ongoing maintenance.
